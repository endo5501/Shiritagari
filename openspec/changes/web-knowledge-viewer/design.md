## Context

Shiritagari は Tauri アプリで、Rust バックエンド + React フロントエンドで構成される。AIの知識は SQLite (`Arc<Mutex<Database>>`) に保存され、`AppState` 経由で共有されている。現在のUIはマスコット表示に特化しており、知識の閲覧機能を持たない。

## Goals / Non-Goals

**Goals:**
- 組み込み HTTP サーバで知識の読み取り専用 Web UI を提供する
- Patterns、Episodes、Profile の3ページ構成
- Episodes と Patterns にページング機能を持たせる
- トレイメニューからワンクリックでブラウザを開ける

**Non-Goals:**
- データの編集・削除機能（将来検討）
- フィルタ・検索機能（将来検討）
- Speculations の表示
- 認証・アクセス制御（localhost 限定で十分）
- SPA フレームワーク（React 等）による Web UI

## Decisions

### 1. HTTP サーバ: axum を採用
- **選択**: axum
- **理由**: Tokio ベースで Tauri の async ランタイムと親和性が高い。Rust エコシステムで最も活発なWebフレームワーク
- **代替案**: actix-web（重い）、warp（メンテナンス停滞気味）、tiny_http（機能不足）

### 2. テンプレートエンジン: askama を採用
- **選択**: askama + askama_axum
- **理由**: コンパイル時にテンプレートを検証でき型安全。axum との統合クレートあり
- **代替案**: tera（動的テンプレート、ランタイムエラーのリスク）、手書き HTML 文字列（保守性低）

### 3. DB アクセス: 既存の `Arc<Mutex<Database>>` を共有
- **選択**: axum の State として既存の DB ハンドルを共有
- **理由**: 追加の DB 接続不要。既存のメソッドを再利用できる
- **注意**: Mutex ロック時間を最小化するため、ロック内でクエリ実行→即解放のパターンを徹底

### 4. ポート戦略: 14789 + 自動探索
- **選択**: 14789 を起点に 100 ポート分を探索
- **理由**: 固定ポートは利便性が高いが、衝突時のフォールバックが必要
- **実装**: `TcpListener::bind` で空きポートを確認、確定したポートを `AppState` に保持

### 5. サーバライフサイクル: アプリ起動時に常時起動
- **選択**: Tauri の setup 内で `tokio::spawn` して常駐
- **理由**: オンデマンド起動より実装がシンプルで、起動時の待ち時間もない

### 6. CSS: テンプレートにインライン埋め込み
- **選択**: base.html の `<style>` タグ内に CSS を記述
- **理由**: 静的ファイル配信の仕組みを追加する必要がなく、1ファイルで完結する。ページ数が3つと少ないため保守性も問題ない

## Risks / Trade-offs

- **[Mutex 競合]** → Web リクエストとポーリングループが同時に DB にアクセスする可能性がある → ロック時間を最小化し、クエリ単位で acquire/release する
- **[ポート衝突]** → 14789 が既に使用中の場合 → 自動探索で回避。ログに実際のポートを出力する
- **[テンプレート変更時の再コンパイル]** → askama はコンパイル時埋め込みのため、テンプレート変更で再ビルドが必要 → 開発体験のトレードオフとして許容（ページ数が少ない）
