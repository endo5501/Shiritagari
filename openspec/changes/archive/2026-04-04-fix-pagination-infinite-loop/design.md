## Context

`poller.rs` の `poll_once()` はイベントをページネーションで取得する。各ページの最後のイベントのタイムスタンプを次ページの `start` パラメータとして使用している。しかし ActivityWatch APIの `start` はinclusive（その時刻のイベントを含む）であるため、同じイベントが繰り返し返され、100件以上のイベントがある場合に無限ループとなる。

## Goals / Non-Goals

**Goals:**
- ページネーションの無限ループを解消する
- ページ間でのイベント重複取得を防止する
- ページネーションに安全上限を設けて想定外の無限ループを防ぐ

**Non-Goals:**
- ActivityWatch APIの仕様変更
- ページネーション以外のイベント取得ロジックの変更

## Decisions

### 1. タイムスタンプに1ミリ秒加算してcursorを進める

**決定**: 次ページのcursorとして `last.timestamp + 1ms` を使用する。

**理由**: ActivityWatch APIの `start` がinclusiveであるため、同じタイムスタンプを使うと前ページの最後のイベントが再度含まれる。1ms加算することで確実にcursorが進む。ActivityWatchのタイムスタンプはISO 8601形式（例: `2026-04-04T01:58:46.123+00:00`）であり、ミリ秒精度で記録されるため、1ms加算で同一ミリ秒内の別イベントをスキップするリスクは実用上無視できる。

**代替案**: イベントIDベースのページネーション → ActivityWatch APIが `after_id` パラメータをサポートしていないため不採用。

### 2. ページネーションに最大ループ回数の上限を設ける

**決定**: ページネーションループに最大50回のイテレーション上限（= 最大5000件）を設定し、超過時はwarningログを出して取得済み分で続行する。

**理由**: 仮にcursor進行ロジックに不具合があっても無限ループを防止する安全策。通常の10分間隔ポーリングで5000件を超えることは考えにくい。

## Risks / Trade-offs

- **同一ミリ秒の複数イベント**: 1ms加算により同一ミリ秒内の後続イベントを取りこぼす可能性がある → ActivityWatchのイベント記録精度を考慮すると実用上問題なし。また下流のイベントID重複排除が二重の安全策として機能する。
- **最大ループ到達時のイベント欠落**: 5000件を超えるイベントは取得されない → warningログで通知し、次回ポーリングでcursorから再開される。
