## Context

Shiritagari は10分ごとのポーリングサイクルで ActivityWatch からイベントを取得し、パターンマッチングと LLM 推論を経て質問を生成する。現在 `log` クレートと `env_logger` が Cargo.toml に含まれているが、`env_logger` の初期化が行われておらず、既存の `info!`/`warn!` マクロ（`poller.rs` に3箇所）も出力されない。メインループ（`lib.rs`）と推論エンジン（`engine.rs`）にはログが一切なく、処理がどこで止まっているか判断できない。

## Goals / Non-Goals

**Goals:**
- `npm run tauri dev` 実行時、ターミナルにポーリングサイクルの判定過程が出力される
- 各判定ポイントで「何が起きて、どう判断したか」が分かる
- `RUST_LOG` 環境変数でログレベルを制御できる（デフォルト: 出力なし）

**Non-Goals:**
- フロントエンド側のデバッグパネル
- ログのファイル出力やログローテーション
- 既存の動作やパフォーマンスへの変更

## Decisions

### 1. `env_logger` の初期化場所

**決定**: `lib.rs` の `run()` 関数冒頭で `env_logger::init()` を呼ぶ。

**理由**: Tauri アプリのエントリーポイントであり、全モジュールのログを有効化する最も早いタイミング。`main.rs` はライブラリの `run()` を呼ぶだけなので `lib.rs` 側で初期化するのが適切。

### 2. ログレベルの使い分け

**決定**:
- `debug!` — サイクル開始/終了、取得イベント数、フィルタ結果など詳細な判定過程
- `info!` — パターンマッチ結果、LLM 呼び出し結果、質問送信など重要な判定結果
- `warn!` — ActivityWatch 不可、プロバイダー生成失敗など注意が必要な状態（既存の `warn!` と整合）

**理由**: `RUST_LOG=info` で判定結果の概要を、`RUST_LOG=debug` で詳細な過程を確認できる二段階の粒度。

### 3. ログの配置箇所

**決定**: 以下の3ファイルにログを追加する。

| ファイル | 追加するログ |
|---------|------------|
| `lib.rs` (メインループ) | サイクル開始、AFK スキップ、イベント空スキップ、プロバイダー生成失敗、パターンマッチ結果分岐、LLM 失敗、質問送信、acknowledge 完了 |
| `poller.rs` (ポーリング) | 取得イベント数、フィルタ後の新規イベント数、カーソル位置 |
| `engine.rs` (推論) | フィルタ前後のイベント数、パターンマッチの詳細（アプリ名、信頼度、アクション）、LLM コンテキスト概要、LLM 結果（confidence、should_ask） |

**理由**: ログを各モジュールに分散させることで、各モジュールが自分の責務に関するログを持つ。メインループは全体の流れ、各モジュールは詳細を担当する。

### 4. ログのプレフィックス規約

**決定**: Rust の `log` クレートは自動的にモジュールパスをターゲットとして使うため、カスタムプレフィックスは不要。`RUST_LOG=shiritagari_app_lib::polling=debug` のようにモジュール単位でフィルタ可能。

**代替案**: `[Poll]`、`[Inference]` 等のカスタムプレフィックスをメッセージに付ける案は、`log` クレートの標準機能と重複するため不採用。

## Risks / Trade-offs

- **ログ出力のパフォーマンスコスト** → `RUST_LOG` 未設定時はマクロが何も出力しないため影響なし。`env_logger` は起動時にフィルタを構築し、ログレベル外のメッセージは即座にスキップされる。
- **ログに機密情報が含まれるリスク** → タイトル情報は既にプライバシーフィルタを通過したもののみ推論に渡されるため、フィルタ後の情報をログに含めても問題ない。ただし LLM のレスポンス全文のログ出力は避け、判定結果（confidence、should_ask）のみ出力する。
