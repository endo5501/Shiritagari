## Context

現在 `poller.rs` の `poll_once()` はAFK検知時にイベント取得をスキップし空の `PollResult` を返す。`lib.rs` のメインループは `skipped_afk` フラグを見て即 `continue` する。このため、ポーリングタイミングとAFKが重なるとモニタリングが長期間行われない。

## Goals / Non-Goals

**Goals:**
- AFK中でもイベント取得と推論を実行し、モニタリングの機会を逃さない
- AFK中に生成された質問をユーザ復帰時に提示する
- 変更を最小限に抑え、既存の重複排除ロジックを活用する

**Non-Goals:**
- AFK復帰のリアルタイム検知（次回ポーリングでの検知で十分）
- 質問の永続化（メモリ上のキューで十分、再起動時に失われてよい）
- 複数質問のキューイング（最新1件のみ保持）

## Decisions

### 1. `poll_once()` からAFKスキップを削除

**決定**: AFK状態に関わらずイベント取得を実行する。`is_afk` フラグは `PollResult` に残すが、`skipped_afk` フィールドは廃止する。

**理由**: AFK前の操作データは推論の価値がある。AFK状態はイベント取得の可否ではなく、質問の提示タイミングの判断に使う。

**代替案**: AFK時にイベント取得だけして推論は遅延する案 → 不採用。推論まで一貫して実行し、結果（質問）の提示だけ遅延する方がシンプル。

### 2. 質問キューは `Option<String>` で実装

**決定**: メインループ内に `let mut pending_question: Option<String> = None;` を持ち、AFK中の質問を上書き保存する。

**理由**: 最新1件のみ保持する方針のため、`Vec` やDB永続化は不要。`Option` の上書きで自然に最新1件が残る。

### 3. AFK復帰検知は次回ポーリングで行う

**決定**: 前回サイクルの `is_afk` を保持し、`was_afk && !is_afk` の条件でキューの質問をemitする。

**理由**: 追加のポーリングループやイベント監視は不要。最大 interval_minutes 分の遅延は許容範囲。

### 4. AFK中でも質問がない場合のemitスキップ

**決定**: AFK復帰時に `pending_question` が `None` であれば何もしない。通常の推論フローで質問が生成されなかった（Silent/パターンマッチ済みなど）場合に対応。

## Risks / Trade-offs

- **AFK中のLLM APIコスト**: AFK中も推論が走るため外部API呼び出しが発生する → 既存の重複排除（カーソル・イベントID）により同じイベントは再処理されない。AFK中に新イベントが発生しない限り推論は走らないためリスクは低い。
- **復帰検知の遅延**: 最大 interval_minutes 分遅延する → 許容範囲として合意済み。
- **再起動時のキュー消失**: メモリ上キューのため再起動で失われる → 質問の永続化は Non-Goal として合意済み。
